# Умные фильтры подписки + наличие мест при подписке

Дата: 2026-06-19
Статус: согласовано, готово к плану реализации

## Контекст и цель

Сейчас подписка отслеживает «любые места» по маршруту/поезду и при создании не
показывает текущее наличие. Пользователь хочет: (1) при подписке сразу видеть,
сколько мест есть и какие; (2) уточнять подписку по типу вагона, полке и цене,
чтобы уведомления приходили только про подходящие места.

API РЖД (`train-pricing → CarGroups`), который бот уже вызывает, отдаёт на уровне
вагона: тип (`CarType`/`CarTypeName`), количество мест по полкам
(`Lower/Upper/LowerSide/UpperSide PlaceQuantity`), цену (`MinPrice/MaxPrice`),
доступность (`AvailabilityIndication`). Признака «у окна» там НЕТ — он только в
схеме вагона (отдельный запрос `car-pricing`, по каждому вагону), что дорого для
мониторинга. Поэтому фильтры вагон/полка/цена строятся на уже доступных данных,
а «у окна» выносится в изолированный best-effort модуль (только просмотр).

## Объём

В объёме:
- Показ разбивки наличия при выборе поезда (до подписки).
- Фильтры подписки: тип вагона (мульти), полка (одиночный), цена-потолок (пресеты).
- Панель тогглов с живым пересчётом «под фильтр: N мест» без лишних запросов.
- Фильтр-осведомлённый мониторинг и уведомления + «Проверить сейчас».
- «У окна» — изолированный best-effort модуль, только при подписке/проверке.

Вне объёма (возможные следующие фазы):
- «У окна» как критерий мониторинга.
- Фильтр по числу мест (min_seats остаётся 1 по умолчанию).
- Ввод цены вручную (используем пресеты).

## Фильтры

| Фильтр | Тип | Значения | Маппинг на данные РЖД |
|---|---|---|---|
| Вагон | мульти | Купе/Плац/СИД/Люкс | `CarType` ∈ {Compartment, ReservedSeat, Sedentary, Soft}; пусто = любой |
| Полка | одиночный | Низ / Верх / Любая | Низ = `Lower+LowerSide`, Верх = `Upper+UpperSide`, Любая = без ограничения |
| Цена | одиночный | ≤3000 / ≤5000 / ≤8000 / Любая | `max_price` (₽), 0 = любая; сравнение с `CarGroup.MinPrice` |

## Модель данных и миграции

`subscriptions`: добавить `berth TEXT DEFAULT 'any'`, `max_price INTEGER DEFAULT 0`.
Тип вагона хранится в существующем `car_types` (CSV кодов `CarType`).
`search_states`: добавить `filter_car_types TEXT DEFAULT ''`,
`filter_berth TEXT DEFAULT 'any'`, `filter_max_price INTEGER DEFAULT 0`, и
`selected_train_cargroups TEXT DEFAULT ''` (JSON CarGroups выбранного поезда — для
живого пересчёта в панели без повторных запросов к РЖД).

Миграции — идемпотентным `ALTER TABLE ... ADD COLUMN` в `init_database()`, по
образцу существующей миграции `search_states.messages_to_delete`
(`database/manager.py`). Обновить INSERT/SELECT и сериализацию `SearchState`/
`Subscription` (`database/models.py`, `database/manager.py`), сохранив текущую
CSV-сериализацию `messages_to_delete`.

## Матчинг мест (единая точка)

В `services/rzd_api.py` (`RZDAPIService`) добавить:

```
match_seats(train, car_types=None, berth='any', max_price=0) -> dict
# {'total': int, 'lower': int, 'upper': int, 'min_price': float|None,
#  'by_type': {CarTypeName: int}}
```

Считает только `CarGroups` с `AvailabilityIndication == 'Available'`, проходящие
фильтры: `CarType` ∈ car_types (если задано); при berth='lower'/'upper' —
соответствующая полка > 0 в группе (учитываем боковые); `MinPrice <= max_price`
(если max_price>0). Реюз существующих `count_seats_breakdown`/`min_price` как
частных случаев. Используется при подписке, в мониторинге и в «Проверить сейчас».

## UX при подписке (`handlers/search.py`)

1. После выбора поезда (`handle_select_train`) — один `train-pricing` через
   `asyncio.to_thread`, найти поезд, сохранить его CarGroups в
   `search_state.selected_train_cargroups`, сбросить фильтры в дефолт.
2. Показать панель (редактируется на месте): строка наличия (всего + по типам +
   низ/верх + от цены) и тогглы:
   - Вагон: `flt_car_<CarType>` (мульти-тоггл)
   - Полка: `flt_berth_<lower|upper|any>` (одиночный)
   - Цена: `flt_price_<3000|5000|8000|0>` (одиночный)
   - `subscribe_filtered`
3. Любой `flt_*` callback: обновляет соответствующее поле в `search_state`,
   пересчитывает «под фильтр: N мест» через `match_seats` из кэша CarGroups
   (без обращения к РЖД), перерисовывает текст+клавиатуру с подсветкой выбранного.
4. `subscribe_filtered`: создаёт `Subscription` с `car_types/berth/max_price`,
   подтверждение содержит сводку фильтра, текущее число подходящих мест и
   deep-link (`build_purchase_url`). Очищает `search_state`.

Сохранить инварианты флоу: единое прогресс-сообщение, `messages_to_delete`,
`save_search_state` после каждой мутации.

## Мониторинг и уведомления (`services/monitoring.py`)

`check_single_subscription`: для каждого поезда считать `match_seats` с фильтрами
подписки; сводка состояния (`current_state`) и решение об уведомлении — по
отфильтрованному числу. Существующая дедуп-логика (`subscription_states`)
сохраняется, но становится фильтр-осведомлённой. В тексте уведомления —
человекочитаемый фильтр («Купе, нижнее, до 8000 ₽») и сколько мест подошло.
Кнопка «🔄 Проверить сейчас» (`check_subscription_now` в `handlers/search.py`)
показывает отфильтрованную разбивку.

## «У окна» — изолированный best-effort модуль

Новый `services/rzd_seatmap.py`: по поезду+вагону вызывает `car-pricing`
(параметры/метод требуют доревёрса — на зонде GET вернул 500) и считает
окна/не-окна. Используется ТОЛЬКО при подписке/«Проверить сейчас», никогда в
цикле мониторинга. До готовности эндпоинта возвращает `None` → в UI «у окна: н/д».
Модуль изолирован: его отказ/отсутствие не влияет на ядро. Если заработает —
строка «у окна: N» в разбивке (информативно, не критерий мониторинга).

## Обработка ошибок

Соблюдаем конвенцию проекта: сервисы/БД логируют и возвращают пустые/дефолтные
значения, не бросают. `match_seats` на кривых данных возвращает нули. Сетевые
вызовы РЖД — через `asyncio.to_thread`. Seatmap-сбой → `None`/«н/д».

## Тестирование

- `match_seats`: комбинации вагон/полка/цена на фикстурах CarGroups (включая
  боковые места, недоступные группы, потолок цены).
- Парсинг/диспатч `flt_*` callback'ов и обновление состояния фильтров.
- Идемпотентность миграций (повторный `init_database` не падает; старые БД
  получают новые колонки).
- Seatmap: парсинг по сохранённой фикстуре (когда эндпоинт известен) + graceful
  `None` при ошибке. Помечается, что мониторинг seatmap не использует.
- Все новые юнит-тесты — не integration; быстрая сюита остаётся зелёной.
